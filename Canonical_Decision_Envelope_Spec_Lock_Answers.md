# Canonical Decision Envelope + Outcome Feedback Loop
## Spec-Lock Answers

Purpose: provide hard, implementation-guiding answers for all open spec questions without redesigning existing subsystems.

---

## 1) Authority and ownership

### 1.1 Authoritative writer of envelope files
Answer: new module under DevFabric.

Locked decision:
- Create a dedicated writer/coordinator module under DevFabric control-plane namespace (for example: decision_envelope_manager.py).
- devfabeco_orchestrator.py invokes it.
- proof_manager.py remains registry/packaging/manifest infra, not semantic decision owner.

### 1.2 Decision ownership model
Answer: ownership varies by stage, with DevFabric as chain authority.

Locked decision:
- Stage producer ownership can vary by stage for provenance.
- Chain authority (ordering, hashing, persistence) is always DevFabric control-plane writer.
- owner_component field is mandatory on each envelope entry.

Owner map:
- preflight: DevFabric
- planning: NGKsGraph
- pre-build: DevFabric
- execution: NGKsBuildCore
- post-build: DevFabric
- certification: DevFabric (with certification subsystem refs)
- rerun-remediation: DevFabric

### 1.3 Envelope scope
Answer: per command invocation plus per target child chains when multi-target.

Locked decision:
- Root scope: one envelope chain per command invocation.
- If multiple targets exist: create child chains per target.
- Preserve both:
  - root chain (global command decisions)
  - child chains (target-specific decisions)

## 2) Run identity and scope

### 2.1 Canonical run_id source
Answer: generated once by DevFabric, with optional user override.

Locked decision:
- Primary: generated by DevFabric at command start.
- Persisted in run manifest and reused by all stage writes.
- User override allowed only via explicit argument/env and must be recorded as run_id_source=user_override.
- Proof-folder naming may include run_id but is not the source of truth.

### 2.2 Rerun/remediation identity
Answer: new run_id linked to parent.

Locked decision:
- Rerun/remediation cycle gets new run_id.
- parent_run_id is mandatory for lineage.
- Same run_id is not reused across independent command invocations.

### 2.3 stage_id flexibility
Answer: fixed enums now + extension mechanism.

Locked decision:
- Core stages locked for v1:
  - preflight
  - planning
  - pre-build
  - execution
  - post-build
  - certification
  - rerun-remediation
- Extensions allowed only via stage_extension namespace and schema version bump.

## 3) Write behavior

### 3.1 File shape
Answer: append-only chain file plus per-stage materialized views.

Locked decision:
- Canonical source: chain file.
- Per-stage files are deterministic views derived from chain entries.

### 3.2 Skipped stages
Answer: always write skipped stage envelopes.

Locked decision:
- Even skipped stages must write entries with:
  - gate_decision=SKIP
  - reason codes
  - missing inputs list
  - hash-chain continuity

### 3.3 Abrupt failure mid-stage
Answer: write INCOMPLETE when possible + recovery envelope on next start.

Locked decision:
- If process can trap failure: write stage envelope with gate_decision=INCOMPLETE.
- If hard crash prevents write: next startup writes recovery envelope with reason=prior_unclosed_stage.
- Never silently omit chain continuity.

### 3.4 Summary markdown authority
Answer: summary markdown is derived only from chain JSON.

Locked decision:
- Markdown is generated view, never independent source.
- Any mismatch between markdown and chain is treated as non-authoritative markdown drift.

## 4) Gate semantics

### 4.1 Allowed gate enum
Locked decision:
- PASS
- FAIL
- BLOCK
- WARN
- ADVISORY
- SKIP
- INCOMPLETE
- INVALID_CHAIN

### 4.2 FAIL vs BLOCK vs DENY vs HALT
Locked decision:
- FAIL: stage completed and criteria not met.
- BLOCK: stage not allowed to start due to upstream gate/policy.
- DENY: reserved alias of BLOCK (do not use as independent enum in v1).
- HALT: process-control action, not gate enum value.

### 4.3 Precedence when policy/root-cause/certification disagree
Locked decision:
1. INVALID_CHAIN (integrity failure of decision chain)
2. POLICY blocking outcome
3. CERTIFICATION hard fail
4. ROOT_CAUSE-derived hard block (only if mapped as blocking in policy matrix)
5. WARN/ADVISORY outcomes

### 4.4 History/predictive authority
Answer: advisory by default.

Locked decision:
- History and predictive signals are advisory only in v1.
- They can hard-block only if explicit policy rule maps their codes to BLOCK.

- DevFabric adapters normalize into strict normalized_findings schema before envelope write.

### 5.2 Mandatory finding fields
Locked decision:
- source_component
- source_artifact
- severity
- reason_code
- confidence
- blocking
- evidence_refs
- timestamp_utc
- stage_name

### 5.3 Reason-code taxonomy
Answer: must be canonically defined in this milestone.

Locked decision:
- Introduce canonical reason-code taxonomy file and version it.
- Map all producer-local codes through adapter mappings.

### 5.4 Confidence format
Locked decision:
- Store raw score as float 0.0-1.0.
- Also store band (LOW|MEDIUM|HIGH).
- Percent is derived view only.

## 6) Feedback-loop behavior

### 6.1 action_taken definition
Answer: includes machine and operator actions.

Locked decision:
- action_executor enum must include:
  - system
  - operator
  - mixed

### 6.2 Manual actions
Answer: represent explicitly.

Locked decision:
- Manual actions are first-class records (examples: profile edit, SDK install, source fix, policy update).

### 6.3 One or many actions per envelope
Answer: many allowed.

Locked decision:
- One envelope can have zero-to-many action records.
- Each action has unique action_id.

### 6.4 Proposed but not taken
Answer: still write feedback record.

Locked decision:
- Required record with status:
  - proposed=true
  - executed=false
  - observed_effect=no_change (unless evidence says otherwise)

### 6.5 Confidence update application
Answer: immediate in feedback artifact; propagated asynchronously to history/predictive stores.

Locked decision:
- Write confidence delta immediately to feedback chain.
- Apply persistent model/store updates in deterministic post-commit step.

## 7) Replay and determinism

### 7.1 Replay target
Locked decision:
- Primary target: identical gate decisions and identical envelope hashes.
- Secondary: regenerate deterministic summaries/markdown from chain.

### 7.2 Replay input boundary
Locked decision:
- Replay reads proof-packet artifacts only.
- Workspace reads are disallowed in strict replay mode.

### 7.3 Missing evidence behavior
Locked decision:
- Do not hard-fail by default.
- Emit missing_input reason codes and degrade deterministically.
- Optional strict replay mode may hard-fail.

### 7.4 Hash inclusion rules
Locked decision:
- Exclude non-semantic fields from hash:
  - timestamp_utc
  - writer_runtime_ms
  - local_path_variants
- Include deterministic semantic payload and refs.

### 7.5 timestamp hashing
Locked decision:
- timestamp_utc is informational only, excluded from semantic hash.

## 10) History / predictive / remediation authority

### 10.1 Automatic confidence updates
Locked decision:
- Allowed, but bounded and rule-driven.
- Every adjustment must include reason code and source evidence refs.

### 10.2 Confidence delta bounds
Locked decision:
- Max absolute delta per feedback event: 0.05
- Max absolute cumulative delta per rerun cycle: 0.10

### 10.3 Recurrence impact format
Locked decision:
- Store both:
  - categorical (reduced/unchanged/increased)
  - numeric score delta

### 10.4 Predictive calibration effect
Locked decision:
- v1 modifies reporting metrics and calibration stats only.
- No automatic model-state mutation in v1.
- Threshold/ruleset changes require explicit policy update path.
