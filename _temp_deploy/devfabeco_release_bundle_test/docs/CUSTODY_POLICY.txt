================================================================================
CUSTODY POLICY: Non-Negotiable Chain of Custody
================================================================================

This bundle enforces the authoritative NGKsDevFabEco execution chain.
Certification and audit claims are valid only when this chain is preserved.

AUTHORITATIVE CHAIN (in order):

1. NGKsEnvCapsule
   Responsible for: Resolving and locking environment state
   Output: env_capsule.resolved.json, env_capsule.lock.json
   
2. NGKsGraph
   Responsible for: Detecting project intent and generating deterministic plan
   Output: graph.json, compile_commands.json, ngksbuildcore_plan.json
   
3. NGKsBuildCore
   Responsible for: Executing the approved plan
   Output: build results, proof artifacts in _proof/<run_id>/
   
4. NGKsLibrary / NGKsDevFabric
   Responsible for: Assembling proof, enforcing certification gates
   Output: certification decision, gate verdict, HTML dashboard

CRITICAL RULES:

✓ External tools (Ninja, CMake, ad-hoc scripts) may be DETECTED
  but are NOT AUTHORITATIVE for execution decisions.

✓ The bundle will NOT:
  - Bypass NGKsBuildCore with direct Ninja execution
  - Run external build scripts outside DevFabEco orchestration
  - Treat foreign artifacts as authoritative gate evidence
  - Weaken existing certification logic
  - Accept proof from out-of-contract runs

✓ If downstream repos replace NGKsBuildCore with Ninja/external execution:
  - That run is OUT-OF-CUSTODY
  - Certification claims are VOID
  - CANNOT be used for release gating

WHAT THIS MEANS FOR YOU:

As an operator using this bundle:
  □ Trust the PASS/FAIL decision from the dashboard
  □ All gates are enforced, cannot be bypassed
  □ Proof is complete and auditable
  □ You can confidently ship on PASS
  □ You can confidently investigate on FAIL

As a developer extending this bundle:
  □ Do NOT add external executor paths
  □ Do NOT weaken gate logic
  □ Do NOT assume tribal knowledge
  □ Do preserve all proof artifacts
  □ Do add runbook documentation for new failure modes

VERIFICATION:

To verify chain of custody is intact:
  1. Open: _proof/<run_id>/99_summary.txt
  2. Check: "Execution chain: confirmed"
  3. Verify: All 4 phases (EnvCapsule, Graph, BuildCore, Library) completed
  4. Trust: The PASS/FAIL decision

If chain is broken:
  □ Run will exit with FAIL
  □ Proof will document why
  □ Certification claims become VOID

NO EXCEPTIONS.

See: certification_workflow.md (in repository root)
