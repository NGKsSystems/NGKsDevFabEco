================================================================================
PROOF LIFECYCLE: Managing Proof Artifacts
================================================================================

When you run the bundle, it generates proof artifacts. This document explains
what they are, how long to keep them, and when you can delete them.

PROOF FOLDER STRUCTURE:
══════════════════════

After each run, a new proof folder appears:

  _proof/
    └─ devfabric_run_20260323_141500/     [One per run]
       ├─ 00_run_header.txt               [Start info, timestamp, commit]
       ├─ 10_envcapsule/                  [Environment details]
       ├─ 20_graph/                       [Dependency graph]
       ├─ 30_buildcore/                   [Build artifacts & logs]
       ├─ 40_library/                     [Proof index & reports]
       ├─ 99_summary.txt                  [Human-readable summary]
       └─ [other metadata files]

HOW LONG TO KEEP PROOF:
══════════════════════

IMMEDIATE (0 days):
  Keep the latest proof if:
  □ You're debugging a FAIL
  □ You're investigating a PASS
  
  Can delete if:
  □ Decision has been acted upon (shipped or fixed)
  □ Proof has been archived

SHORT-TERM (1-7 days):
  Keep while:
  □ The build is actively being developed
  □ You might need to re-reference the result
  □ Certification discussions are ongoing
  
  Action: Archive or delete after release

MEDIUM-TERM (1 month):
  Keep if:
  □ You're tracking proof for an official release
  □ You need regression baseline comparisons
  □ Audit/compliance requirements mandate it
  
  Action: Archive to external storage or delete

LONG-TERM (+1 month):
  Keep only if:
  □ Regulatory/compliance requirements (rare)
  □ Historical forensics needed
  
  Action: Archive to cold storage or delete

ARCHIVAL STRATEGY:
══════════════════

Option 1: Zip and Move
  1. cd _proof
  2. Compress-Archive devfabric_run_20260323_141500 devfabric_run_20260323_141500.zip
  3. Move-Item devfabric_run_20260323_141500.zip D:\archive\
  4. Remove-Item devfabric_run_20260323_141500 -Recurse

Option 2: Delete (if not needed)
  1. cd _proof
  2. Remove-Item devfabric_run_20260323_141500 -Recurse

Option 3: Keep as-is (for intra-sprint tracking)
  □ No action needed
  □ _proof folder is git-ignored (won't bloat repo)

WHAT INSIDE PROOF TO KEEP:
══════════════════════════

If you delete entire folders, at minimum keep:
  ✓ 00_run_header.txt        (metadata: commit, timestamp, env)
  ✓ 20_graph/                (requirement trace)
  ✓ 99_summary.txt           (human summary)
  ✓ 40_library/              (certification decision)

You can delete:
  ~ 10_envcapsule/ (environment details, bulky, reproducible)
  ~ 30_buildcore/  (build logs, very bulky, reproducible)
  ~ [other metadata]

PROOF SIZE WARNINGS:
════════════════════

Large proof folders occur when:
  □ Many source files (1000+): 50-100 MB per run
  □ Large binaries generated: 100-500 MB per run
  □ Deep build graphs: 10-50 MB
  □ Many test artifacts: 50-200 MB

If disk space is low:
  1. Identify largest proof folders: dir _proof | sort-object
  2. Check what's worth keeping
  3. Archive or delete old runs
  4. Keep latest 2-3 runs only

CLEANUP SCRIPT EXAMPLE:

# Delete proof folders older than 30 days
Get-ChildItem _proof -Filter "devfabric_run_*" -Directory | 
  Where-Object {$_.LastWriteTime -lt (Get-Date).AddDays(-30)} |
  Remove-Item -Recurse

# Archive latest run
$Latest = Get-ChildItem _proof -Filter "devfabric_run_*" -Directory | 
  Sort-Object LastWriteTime -Descending |
  Select-Object -First 1
Compress-Archive $Latest "$($Latest.Name).zip"

RETENTION MATRIX:
══════════════════

Release Status    | Desktop      | CI Server   | Archive
───────────────────────────────────────────────────────
Development       | Keep 1-2     | Keep 3-5    | Delete after release
Release Candidate | Keep 1       | Keep Latest | Keep Latest + 2 prev
Released (Stable) | Keep Latest  | Keep Latest | Archive to cold storage
Regression Found  | Keep All     | Keep All    | (Forensics important)

RECOVERY:

If you deleted a proof folder and need it back:
  □ Check git history: git log --all
  □ Check backup system: (ask your IT team)
  □ If gone: Recreate by re-running the bundle

You cannot recover deleted proof. Archive before deleting.

SUMMARY:
════════

□ Proof is large and accumulates
□ Keep latest 1-3 runs actively
□ Archive old runs if compliance requires
□ Delete freely after release (no compliance) or
□ Keep per your organization's retention policy
□ Proof is git-ignored, won't bloat repository
